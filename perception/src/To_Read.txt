OVERVIEW
--------
This subsystem detects letter blocks in the workspace using an Intel RealSense
D435i RGB-D camera and a custom-trained CNN. It classifies each block, computes
its 3D position, and publishes that information to the rest of the system.

Two coordinate outputs are produced:
  - Camera frame coordinates  ->  /perception/detections  (Kermit, Elijah)
  - World frame coordinates   ->  /perception/gameboard_state  (Connor)

The world frame transform is computed via the UR3e TF tree at runtime — no
manual coordinate conversion is required.


================================================================================
HARDWARE DEPENDENCIES
================================================================================

  - Intel RealSense D435i camera (USB 3.0)
      Mounted on the UR3e end-effector, approximately 5 cm back and 5 cm above
      the gripper_tcp (tool centre point). Exact mount offsets are configured
      via calibration variables at the top of realsense_camera_cnn.py.

  - UR3e robotic arm with OnRobot RG2 gripper
      Must be running and connected before the perception node attempts any
      camera-to-world coordinate transforms.

  - Letter blocks
      5 cm x 5 cm wooden blocks with white paper labels glued to the face.
      The white paper face is what the camera detects — CARD mode is used
      because it finds white regions via brightness threshold, which works
      correctly with this block design. The depth sensor reads the paper
      surface to give the 3D position of each block.


================================================================================
DETECTION MODE EXPLAINED
================================================================================

  DETECTION_MODE = "CARD"   <-- correct setting for white-paper-on-block setup

  CARD mode uses a white brightness threshold to find block faces. Any region
  brighter than CARD_BRIGHTNESS (180/255) is treated as a potential block face.
  This works because the white paper labels stand out clearly against the
  workspace surface. The inner 90% of each detected region is then fed to the
  CNN (CARD_MARGIN = 0.10 trims 10% from each edge to exclude the paper border).

  BLOCK mode (not used) would use adaptive thresholding and a depth gate — this
  was designed for bare wooden blocks without paper labels. Not needed here.

================================================================================
HL_CONTROL — WHAT IT IS AND WHY YOU NEED TO BUILD IT
================================================================================

  ROS2 uses strongly typed messages — every topic has a defined message type
  that both the publisher and subscriber must agree on exactly. Standard types
  like Bool, String, and Image come pre-built with ROS2. But Connor's
  hl_control package defines two custom message types that don't exist in
  standard ROS2:

    hl_control/GameboardState   — a list of letter blocks with poses
    hl_control/LetterObject     — a single block: letter + object_id + pose

  These are defined in:
    motion_planning_and_control/hl_control/msg/GameboardState.msg
    motion_planning_and_control/hl_control/msg/LetterObject.msg

  Because they are custom types, ROS2 needs to generate the Python and C++
  code for them before any node can import or use them. That generation happens
  when you run:

    colcon build --packages-select hl_control

  After building, the types become importable in Python:

    from hl_control.msg import GameboardState, LetterObject

  This is why the perception node does a try/except import at startup:

    try:
        from hl_control.msg import GameboardState, LetterObject
        HL_CONTROL_AVAILABLE = True
    except ImportError:
        HL_CONTROL_AVAILABLE = False

  If hl_control hasn't been built, the import fails gracefully — the node
  still starts and all other topics work normally. It just logs a warning and
  skips the /perception/gameboard_state publisher. Connor won't receive world
  frame coordinates until hl_control is built.

  To build it (run from your ROS2 workspace root, not perception/src):

    cd ~/your_ws
    colcon build --packages-select hl_control
    source install/setup.bash

  You only need to do this once per machine, or again if the message
  definitions change.


================================================================================
WHAT "LATCHED" MEANS
================================================================================

  In ROS2, a normal publisher sends a message and it is gone. If a subscriber
  starts up after that message was sent, it misses it and has to wait for the
  next one. For topics that publish continuously (like camera frames or status
  strings), this is fine — the next message arrives a fraction of a second
  later anyway.

  But /perception/gameboard_state is published ONCE per scan — there is no
  "next message" coming soon. If Connor's hl_control_node starts up a few
  seconds after the scan completes, it would miss the gameboard state entirely
  and never know where the blocks are.

  Latched (TRANSIENT_LOCAL) QoS solves this. A latched publisher stores its
  last message. When a new subscriber connects — even minutes later — it
  immediately receives that stored message as if it had been there all along.

  In the code this is set up as:

    LATCHED_QOS = QoSProfile(
        depth=1,
        durability=DurabilityPolicy.TRANSIENT_LOCAL,
        reliability=ReliabilityPolicy.RELIABLE,
    )
    self.pub_gameboard = self.create_publisher(
        GameboardState, '/perception/gameboard_state', LATCHED_QOS)

  Connor's subscriber also uses TRANSIENT_LOCAL so both ends agree on the
  durability policy — this is required in ROS2. If only one end is latched,
  the message is not delivered.

  The same principle applies to /tf_static — the static TF the perception
  node broadcasts at startup (gripper_tcp -> camera_color_optical_frame) is
  also latched, which is why any node that starts later can still look up
  that transform immediately.

  Summary:
    Normal topic  = publish and forget, subscriber must be ready
    Latched topic = publish and remember, subscriber gets it whenever it connects

================================================================================
DURING A SCAN — WHAT TO EXPECT
================================================================================

  1. Press SPACEBAR in the display window to enter AT POSITION mode.

     Terminal 3 logs:
       [INFO] [realsense_cnn_node]: [Trigger] -> AT POSITION

     Display window HUD changes from "MOVING - STANDBY" to "AT POSITION - READING".

  2. Hold letter blocks in front of the camera. After FRAMES_TO_AVERAGE (15)
     consistent frames, each block is reported.

     Terminal 3 logs (every ~1 Hz, only when blocks detected):
       [INFO] [realsense_cnn_node]: [Detections]
         A 94%  cam=(0.041, -0.023, 0.382)  world=(0.183, 0.312, 0.025)  theta=5.0deg
         B 87%  cam=(-0.031, -0.019, 0.391)  world=(0.102, 0.318, 0.024)  theta=-3.1deg

     /perception/detections topic publishes continuously at ~30 Hz.
     Terminal only logs once per second to avoid flooding — topic is faster.

  3. Press SPACEBAR again to leave AT POSITION (or send IDLE on /mission/state).

     Terminal 3 logs:
       [INFO] [realsense_cnn_node]: [Trigger] -> MOVING
       [INFO] [realsense_cnn_node]: [GameboardState] Published 2 blocks in world frame

     /perception/gameboard_state is published once with all detected blocks
     in world frame coordinates. Connor's hl_control_node receives this.

  4. If a human enters the frame:

     Display window shows "HUMAN DETECTED" overlay text.
     /perception/human_detected publishes True at ~5 Hz.
     Elijah's behaviour tree receives this and can pause the mission.
     No terminal log line for human detection — visual only.

================================================================================
INTEGRATION WITH GAMIFICATION NODE (Kermit — SS4)
================================================================================

  The gamification node subscribes to /perception/detections and uses the
  block positions to populate its placement JSON (x_m, y_m, z_m per letter).

  For detections to reach the gamification node, ALL of the following must be
  true at the same time:

    1. Perception node is AT POSITION (SPACEBAR pressed or SCANNING received)
    2. Blocks are visible and CNN confidence >= CNN_CONF_THRESHOLD
    3. Gamification node has game_active = True
    4. Gamification node has game_mode = 'A' and mode_locked = True
    5. Gamification node has guess_pending = True

  guess_pending is set True when a new round starts (after secret word accepted,
  or after feedback). It is set False once a guess is published, preventing the
  node from re-guessing on every incoming detection frame.

  Correct startup sequence for a full Mode A game:

    Step 1 — Set mode:

      COMMAND:
        ros2 topic pub --once /gamification/mode std_msgs/String "data: 'MODE_A'"

      EXPECTED OUTPUT (gamification terminal):
        [INFO] [gamification_node]: Game mode set to Mode A.
        [INFO] [gamification_node]: Published diagnostics: SELECT_SECRET

    Step 2 — Set secret word:

      COMMAND:
        ros2 topic pub --once /gamification/secret_word std_msgs/String "data: 'crane'"

      EXPECTED OUTPUT (gamification terminal):
        [INFO] [gamification_node]: Mode A secret word accepted.

    Step 3 — Start game:

      COMMAND:
        ros2 topic pub --once /mission/state std_msgs/String "data: 'START'"

      EXPECTED OUTPUT (gamification terminal):
        [INFO] [gamification_node]: [Mission] Mode A game started.
        [INFO] [gamification_node]: Published diagnostics: ACTIVE

      guess_pending is now True. The node is waiting for detections.

    Step 4 — Put perception AT POSITION with blocks visible:

      Press SPACEBAR in the perception display window.

      EXPECTED OUTPUT (gamification terminal, within ~1 second):
        [INFO] [gamification_node]: Letters available: ['A', 'B', 'C', 'R', 'E', ...]
        [INFO] [gamification_node]: Attempt 1: Guessing CRANE
        [INFO] [gamification_node]: Published guess: CRANE
        [INFO] [gamification_node]: Published mission state: {"word": "CRANE",
          "attempt": 1, "placements": [{"position": 1, "letter": "C",
          "x_m": 0.183, "y_m": 0.312, "z_m": 0.025, "theta_deg": 5.0}, ...]}
        [INFO] [gamification_node]: Published word request to HL control: CRANE
        [INFO] [gamification_node]: Published diagnostics: GUESSING

      If x_m/y_m/z_m are still null in the published mission state:
        The detections JSON contains null depth — blocks have no depth reading.
        Verify by running: ros2 topic echo /perception/detections
        If x_m is null there too, check blocks are 10-80 cm from the camera
        and that align_depth.enable:=true was passed in Terminal 1.

    Step 5 — Provide feedback after the robot places the word:

      COMMAND:
        ros2 topic pub --once /gamification/feedback std_msgs/String "data: 'GBIBI'"

      (G=green/correct position, B=black/not in word, I=in word wrong position)

      EXPECTED OUTPUT (gamification terminal):
        [INFO] [gamification_node]: Feedback received: GBIBI
        [INFO] [gamification_node]: Candidates remaining: 47

      guess_pending is reset to True. Node waits for the next detections scan.
      Repeat from Step 4.

    To reset between games:

      COMMAND:
        ros2 topic pub --once /mission/state std_msgs/String "data: 'RESET'"

      EXPECTED OUTPUT (gamification terminal):
        [INFO] [gamification_node]: Published diagnostics: RESET
        [INFO] [gamification_node]: [Mission] Game reset.

      Then repeat from Step 1.


================================================================================
CONFIGURATION (top of realsense_camera_cnn.py)
================================================================================

  DETECTION_MODE = "CARD"
      Correct setting for white-paper-on-block setup. Do not change.

  CNN_CONF_THRESHOLD = 32.0
      Minimum confidence % to report a letter.
      Lower = more guesses. Higher = fewer false reads.

  FRAMES_TO_AVERAGE = 15
      Temporal smoothing window. CNN must agree across this many frames
      before reporting a letter. Reduces single-frame misreads.

  CARD_BRIGHTNESS = 180
      Brightness threshold (0-255) for white paper detection.
      Increase if background is bright. Decrease if labels are not detected.

  CARD_MARGIN = 0.10
      Fraction trimmed from each edge of detected white region before
      passing to CNN. Removes paper border. 0.10 = 10% per edge.

  -- Camera intrinsics (replace with values from camera_info topic) ----------
  CAM_FX = 615.0
  CAM_FY = 615.0
  CAM_CX = 320.0
  CAM_CY = 240.0

  To get exact values for your camera:
    ros2 topic echo /camera/camera/color/camera_info --once
  Look for: K: [fx, 0, cx, 0, fy, cy, 0, 0, 1]

  -- Camera mount calibration (relative to gripper_tcp) ----------------------
  CAM_MOUNT_X =  0.00   metres, left/right from gripper_tcp centreline
  CAM_MOUNT_Y =  0.05   metres, upward from gripper_tcp (5 cm up)
  CAM_MOUNT_Z = -0.05   metres, back toward wrist (5 cm back)

  CAM_MOUNT_ROLL  = 0.0   radians — tune these until world coords align
  CAM_MOUNT_PITCH = 0.0   radians   with real block positions in RViz
  CAM_MOUNT_YAW   = 0.0   radians

  Edit values and restart the node to apply. No other code changes needed.

              "letter":    "A",
              "conf":      94.2,
              "x_m":       0.04,    <- camera frame, metres

              "z_m":       0.38,    <- depth from lens, metres

        }

      Published ONCE when scanning ends (IDLE mission state or spacebar off).
      WORLD FRAME coordinates — full camera->world TF applied.
      Contains one LetterObject per detected block:
        letter    : "A"
        object_id : "A_object_1"   (two A blocks -> A_object_1, A_object_2)
        pose      : PoseStamped, frame_id="world"
      All detected blocks with valid depth are included in one message.
      If 5 blocks are detected, all 5 world frame positions are published.
      Subscriber: Connor's hl_control_node (SS2)
      Latched: Connor receives this even if his node starts after publish.
      Requires: colcon build --packages-select hl_control

  /tf_static                      tf2_msgs/TFMessage  (LATCHED)
      One static transform published at startup:
        parent: gripper_tcp
        child:  camera_color_optical_frame
      Adds the final link to the TF chain for camera->world conversion.

================================================================================
COORDINATE FLOW (how a pixel becomes a world coordinate)
================================================================================

  1. RealSense produces pixel (px, py) and depth (metres) for each block centre

  2. pixel_to_camera_frame(px, py, depth):
       x_cam = (px - 320) x depth / 615
       y_cam = (py - 240) x depth / 615
       z_cam = depth
     Result: block position relative to camera lens, in metres.
     Published in /perception/detections (camera frame, every scan frame).

  3. _camera_to_world(x_cam, y_cam, z_cam):
       TF2 looks up: camera_color_optical_frame -> world
       Chain used:   camera -> gripper_tcp -> wrist -> ... -> base_link -> world
       Applies:      p_world = R x p_cam + t
     Result: block position in robot world frame, in metres.
     Published in /perception/gameboard_state (world frame, once per scan).


================================================================================
CNN MODEL
================================================================================

  Architecture:   3x Conv2d (32->64->128 filters) + 2x Linear
                  Input: greyscale 64x64
  Classes:        36 (A-Z, 0-9)
  Training data:  ~400-500 images per class, collected with
                  collect_training_data.py under lab lighting conditions
  Test accuracy:  ~96-97%
  Model file:     perception/outputs/letter_cnn.pt

  To retrain:
    cd perception/src
    python3 collect_training_data.py   <- collect new images
    python3 train_letter_cnn.py        <- train and save model

  Key augmentations applied during training:
    RandomRotation +-20 deg, ColorJitter, RandomPerspective, RandomAffine
    Histogram equalisation (cv2.equalizeHist) applied at both collection
    and inference time to handle lab overexposure.


================================================================================
TROUBLESHOOTING
================================================================================

  "WARNING: letter_cnn.pt not found"
      Run train_letter_cnn.py first, or check the MODEL_PATH variable.
      The node runs in fallback mode showing "?" for all detections.

  "[TF] camera->world lookup failed"
      Robot driver (Terminal 2) is not running. Start it first.
      The node will keep trying — once the driver starts, lookups will work.
      gameboard_state falls back to camera frame coordinates until then.

  "hl_control package not found" warning at startup
      Run: colcon build --packages-select hl_control
      Then: source install/setup.bash
      Then restart the perception node.

  Blocks detected but labelled "?"
      CNN confidence is below CNN_CONF_THRESHOLD (32%). Options:
        - Check lighting (equalizeHist should handle most cases)
        - Lower CNN_CONF_THRESHOLD temporarily to diagnose
        - Retrain the model with more varied data for that class

  gameboard_state published with frame_id = camera_color_optical_frame
      TF was unavailable at time of publish — robot driver was not running.
      World coordinates were not applied. Restart scan with robot driver up.

  camera_color_optical_frame missing from view_frames output
      Static TF broadcast failed or perception node is not running.
      Check log line at startup: "[TF] Static TF broadcast: gripper_tcp -> ..."

  World coordinates are offset from real block positions
      CAM_MOUNT_ROLL/PITCH/YAW need calibration. Place a block at a known
      world coordinate, scan it, compare published values to actual position,
      adjust RPY values at the top of realsense_camera_cnn.py and restart.

  White paper labels not being detected
      Increase CARD_BRIGHTNESS if background is too bright.
      Decrease CARD_BRIGHTNESS if labels are being missed.
      Check MIN_BLOCK_AREA — if blocks appear small in frame, lower this value.

  Gamification node shows x_m: null in placement JSON
      Detections reaching gamification but depth is null.
      Verify by running: ros2 topic echo /perception/detections
      If x_m is null in the raw topic, the depth stream has no reading for
      those blocks. Ensure blocks are 10-80 cm from the camera and that
      align_depth.enable:=true was passed in Terminal 1.

  Gamification node receives no detections (no "Letters available" log line)
      The node silently ignores detections unless all five conditions are met:
        - game_active = True       -> send START on /mission/state
        - game_mode = 'A'          -> send MODE_A on /gamification/mode first
        - mode_locked = True       -> set automatically when MODE_A received
        - guess_pending = True     -> set after secret word accepted or feedback
      Correct order: MODE_A -> secret_word -> START -> scan blocks.
      After a reset, send MODE_A and secret_word again before START.

  ros2 node list stalls / hangs
      Kill and restart the daemon:
        ros2 daemon stop
        ros2 daemon start
        ros2 node list
      If still stalling with cross-machine setup, confirm ROS_DOMAIN_ID matches

================================================================================
LIMITATIONS
================================================================================

  CNN / DETECTION
  ---------------
  - White paper labels required. CARD mode detects block faces by brightness
    threshold — different label materials, colours, or surfaces will break
    detection. The system is not general-purpose object detection.

  - Lab lighting only. The model was trained under lab conditions with
    histogram equalisation applied. Significantly different lighting
    environments (outdoor, strong shadows, coloured ambient light) will
    degrade classification accuracy even with equalizeHist active.

  - Single face per block. If a block is tilted or the wrong face is
    presented to the camera, it will not be detected or classified.

  - No occlusion handling. Overlapping or touching blocks will either
    merge into a single detection or be missed entirely. Blocks must be
    separated in the workspace.

  - Theta is a placeholder. Block rotation (theta_deg) is estimated from
    cv2.minAreaRect and has a 180 degree ambiguity — it cannot distinguish
    a block at 0 degrees from one at 180 degrees. Dot-based disambiguation
    (corner dot = top-left of block face) is the intended solution but is
    not yet implemented.

    - Maximum 5 blocks per scan. MAX_BLOCKS = 5 is hardcoded in the
    detection pipeline. If more than 5 blocks are present in the workspace,
    only the 5 leftmost detections (sorted by x pixel position) are
    returned. Additional blocks are silently ignored.

  DEPTH / POSE
  ------------
  - 4DoF only. Published pose is (x, y, z, theta) — roll and pitch are
    not estimated. The system assumes all blocks are lying flat on the
    workspace surface.

  - Depth reads the paper surface, not the block base. z_m reflects the
    top face of the block. Downstream consumers (Connor's hl_control) must
    account for block height when computing grasp positions.

  - World frame accuracy depends on mount calibration. CAM_MOUNT_X/Y/Z
    and CAM_MOUNT_ROLL/PITCH/YAW have not been physically verified against
    the real robot. World coordinates will be offset until these values are
    calibrated against a known reference point in the workspace.

  INTEGRATION
  -----------
  - Snapshot only. /perception/gameboard_state is published once when
    scanning ends. If a block moves during robot execution, the published
    state becomes stale — there is no automatic re-scan or continuous
    position tracking during pick-and-place.

  - World frame requires robot driver. Without Terminal 2 running,
    TF lookups fail and gameboard_state falls back to camera frame
    coordinates. Perception still functions but Connor's hl_control cannot
    use the output for robot motion.

  - Single camera, single viewpoint. There is no second camera or
    alternative viewpoint to recover from bad angles or blind spots. If the
    robot arm occludes a block from the camera's perspective, that block
    will not be detected until the arm moves.

  STEP 6 — Calibrate camera mount (first real-robot session)
  -----------------------------------------------------------
  World frame coordinates will be offset until CAM_MOUNT_* values are tuned.

  Procedure:
    1. Place a single block at a known position in the workspace
       (measure its x, y from the robot base with a ruler).
    2. Run a full scan — note the world x, y published on
       /perception/gameboard_state.
    3. Compare published values to measured values.
    4. Adjust CAM_MOUNT_X/Y/Z and CAM_MOUNT_ROLL/PITCH/YAW at the top of
       realsense_camera_cnn.py to reduce the error.
    5. Restart the perception node and repeat until error is < 5 mm.

  This only needs to be done once per physical camera mount configuration.
  If the camera is remounted or the gripper is changed, recalibrate.
